Valet runs on language model providers — Anthropic, OpenAI, and Google. Every task you run consumes model usage from one of those providers. This page explains how that usage is paid for, what error messages mean, and who to ask when something is blocked.

## How usage is paid for

Valet calls model providers using API keys configured at the **org level**. When you run a task:

1. Valet picks a model (Anthropic Claude, OpenAI GPT, Google Gemini, or another approved provider).
2. The task uses the org's API key for that provider.
3. Usage shows up on your org's bill or credit balance with that provider.

The org admin is responsible for setting up provider keys and keeping credits or billing active. Individual users do not need to bring their own keys for normal use.

For details on which providers are approved, see the [admin setup](/product/admin-setup) page.

## "Credit balance is too low" or "quota exceeded"

If a task fails with a message like "credit balance is too low," "insufficient quota," "exceeded quota," "payment required," or "billing not active," the problem is with the **org's account at the model provider**, not with your individual user.

What it means in practice:

| Error wording | What is usually happening |
| --- | --- |
| Credit balance is too low | The org has prepaid credits on file with the provider and they have run out. |
| Insufficient quota / exceeded quota | The org's plan with the provider has a usage cap that has been reached. |
| Rate limit / too many requests | The provider is throttling traffic. Often clears in a few seconds; if it persists, the org may need a higher tier. |
| Billing not active / payment required | The org's billing on the provider account is paused or unpaid. |
| Invalid or unauthorized API key | The configured key has been revoked or rotated and Valet needs the new one. |

In all of these cases, the fix is on the provider account, not in Valet. Contact your **Valet admin** — they manage the keys and the billing relationship with the provider.

## "Do Codex / Claude credits come out of my personal seat?"

In Valet's standard configuration, no. Model usage is billed to the org's provider accounts, not to individual user seats. If you have your own personal Claude or ChatGPT subscription, that subscription is separate from how Valet calls the API.

There can be exceptions in custom setups — for example, if your org has chosen to route some users through a personal or per-user provider account. If you are unsure how your org is configured, ask your admin.

## Sandbox and platform usage

Model API calls are not the only cost. When Valet starts a [child session](/product/child-sessions) for code or long-running work, that workspace also uses compute time on Valet's platform. Like model usage, sandbox usage is tracked at the org level. Sessions that finish their work and report back free up that capacity for the next task.

You do not need to manage sandboxes directly. If a session is taking too long or no longer needed, ending it in the task thread is enough.

## What to do when usage is blocked

If you hit a credit or quota error in the middle of a task:

1. Note the exact error message and which model was in use.
2. Ask your Valet admin to check the relevant provider account.
3. Once the admin fixes the underlying balance, retry the task — Valet does not need to be restarted.

If the same task keeps failing the same way, see [troubleshooting](/product/troubleshooting) for general recovery steps.

## For admins

If you are the admin for your org, the provider keys and limits you set determine what every user can do. Practical guidance:

- Use business-tier accounts at each provider so usage is not used for model training.
- Keep at least one fallback provider configured so a single outage does not stop all tasks.
- Watch for low-balance and rate-limit signals proactively. Valet surfaces these as task errors, but the underlying alerts come from the provider.
- See [admin setup](/product/admin-setup) for how org-level keys are configured.
