# Node-first `@valet/api` Implementation Plan

> Supersedes `2026-05-06-worker-to-api-engine-integration.md`. We're not adapting the legacy worker; we're greenfielding a new API package that serves only the agent loop, runs on Node, and uses the portable engine + Docker sandbox + sqlite store directly. The CF worker stays at `packages/worker` for prod and is untouched.

**Goal:** A new `packages/api` package that boots on Node, serves the existing client UI for the agent loop end-to-end (chat, send a prompt, see the agent run `bash` inside Docker, stream events back), and stops there. Anything else 501s.

**Scope discipline:** the hardest job is saying no. Workflows, integrations, mailbox, dashboards, OAuth, channels, telegram/slack â€” out of scope. We add them back later if and only if we want them.

---

## Architecture

```
packages/
  api/                     # NEW â€” Node-first, agent-loop-only
    src/
      main.ts              # Hono + node-server entry
      app.ts               # createApp(providers) â€” composable
      providers/
        types.ts           # Providers interface
        node.ts            # buildNodeProviders({ ... })
        blob-fs.ts         # filesystem BlobStore
      engine/
        host.ts            # EngineHost â€” per-process Map<sessionId, Engine>
        bridge.ts          # engine BusEvent â†’ client WS payload mapper
      routes/
        auth.ts            # GET /api/auth/me (stub)
        sessions.ts        # CRUD on agent sessions
        threads.ts         # list threads
        messages.ts        # POST prompt
        ws.ts              # WS event stream
      schema/              # Drizzle: orgs, users, sessions, threads, messages
      lib/
        drizzle.ts         # AppDb type, migrate helper
        wire.ts            # Engine â†” client message shapes
      middleware/
        auth.ts            # VALET_LOCAL_AUTH=1 stub
        providers.ts       # injects Providers into c.var
    migrations/             # generated by drizzle-kit
  worker/                  # untouched legacy CF worker (prod)
  engine/                  # @valet/engine â€” portable
  store-sqlite/            # @valet/store-sqlite â€” portable
  sandbox-docker/          # @valet/sandbox-docker â€” portable
  shared/                  # @valet/shared types
  client/                  # UI; consumed via VITE_API_URL pointed at Node
```

Key choices:

- **Drizzle + better-sqlite3 only.** No D1 in this package. No D1 types. No D1 imports. No emulator.
- **One sqlite file, two schema sets co-located.** The engine's `SqliteSessionStore` already manages its own tables (`engine_sessions`, `engine_threads`, etc). Our app schema (`agent_sessions`, `users`, `orgs`, `messages`) coexists. Same db file, no conflict â€” table names don't collide.
- **No DOs.** `EngineHost` keeps a `Map<sessionId, Engine>` in process. Sessions are singletons per process; multi-process scaling is out of scope (single-user local dev).
- **Stub auth from the start.** No JWT, no OAuth, no admin middleware. `VALET_LOCAL_AUTH=1` returns a hardcoded user. Anything that needs real auth is out of scope.
- **WebSocket via `@hono/node-ws`.** The wire format matches what the existing UI's `use-chat` hook expects (read `packages/client/src/hooks/use-chat.ts` for the discriminated union of message types).

## Wire format â€” what the client expects

The agent-loop-critical message types from `use-chat.ts`:

| Type | Direction | Payload |
|---|---|---|
| `init` | server â†’ client | `{ session: { id, status, workspace, messages[] }, data: { connectedUsers, runnerConnected, ... } }` |
| `message` | server â†’ client | `{ data: { id, role, content, parts?, threadId?, createdAt } }` |
| `message.updated` | server â†’ client | `{ data: { id, role, content, parts?, createdAt } }` |
| `chunk` | server â†’ client | `{ content, messageId? }` (token streaming) |
| `status` | server â†’ client | `{ status: SessionStatus, data? }` |
| `agentStatus` | server â†’ client | `{ status: 'idle'\|'thinking'\|'tool_calling'\|'streaming'\|'error', detail? }` |
| `error` | server â†’ client | `{ messageId, error?, content? }` |
| `messages.removed` | server â†’ client | `{ messageIds: string[] }` |

Things we explicitly DO NOT implement in v1:
`diff`, `gitState`, `pr.created`, `filesChanged`, `childSession`, `reviewResult`, `title`, `auditLog`, `commandResult`, `modelSwitched`, `toast`, `integrationAuthRequired`, `interactive_prompt`, `interactive_prompt_resolved`, `interactive_prompt_expired`, `models`. The UI will render whatever it gets and ignore unknowns; missing types just means the UI won't show those features.

The client also speaks back over WS: `prompt`, `pause`, `resume`, etc. â€” we'll bridge `prompt` to `engine.session.prompt(...)` and ignore the rest with a 501-like ack.

## What each provider boots from

```ts
buildNodeProviders({
  dbPath: ~/.valet/app.db,
  blobsRoot: ~/.valet/blobs/,
  encryptionKey: process.env.VALET_ENCRYPTION_KEY ?? 'dev-key',
}) {
  // 1. Open better-sqlite3
  const sqlite = new Database(dbPath);
  sqlite.pragma('journal_mode = WAL');
  sqlite.pragma('foreign_keys = ON');

  // 2. Apply app migrations
  applyAppMigrations(sqlite);

  // 3. Apply engine migrations (from @valet/store-sqlite)
  applyEngineMigrations(sqlite);

  // 4. Seed dev user/org
  seedLocalUser(sqlite);

  // 5. Build Drizzle
  const db = drizzle(sqlite, { casing: 'snake_case' });

  // 6. Build providers
  const engineStore = new SqliteSessionStore(db);  // or fresh drizzle-better-sqlite3 instance, depending on the package's API
  const blobs = new FsBlobStore(blobsRoot);
  const sandboxProvider = new DockerSandboxProvider();
  const eventBus = new InMemoryEventBus();
  const engineCredentials = new InMemoryCredentialStore();

  // 7. EngineHost (depends on the four above)
  const engineHost = new EngineHost({ engineStore, sandboxProvider, eventBus, engineCredentials });

  return { db, blobs, encryptionKey, engineStore, sandboxProvider, eventBus, engineCredentials, engineHost };
}
```

Open question to verify when reading `@valet/store-sqlite/src/store.ts`: is `SqliteSessionStore` constructed with a `BetterSQLite3Database` (Drizzle) or a raw `Database`? Read first; pass the right thing. The contract test in `packages/store-sqlite/test/sqlite-store.test.ts` is the reference.

## Known wrinkles

1. **Engine's session/thread/entry vs app's session/thread/message.** Two distinct concepts. The engine owns prompt loop state; the app owns user-visible chat history + session metadata. Don't conflate. The agent's `Session` ID is the app session's ID â€” that's the link.

2. **Wire bridging is glue, not architecture.** Map engine `BusEvent`s to UI message types one type at a time. If the engine doesn't emit something the UI needs, fix it in `bridge.ts` (e.g. issue an `agentStatus` whenever a turn starts/ends).

3. **MessagePart shape.** The UI expects `parts` (text, tool_use, tool_result, image). Engine entries already have a parts-like structure. Mapping should be 1:1 or close.

4. **WebSocket auth.** Existing client opens WS with `['valet', 'bearer.<token>']` subprotocol. In local-auth mode, accept any bearer or none. Don't validate.

5. **Container cleanup.** SIGINT handler must destroy all sandboxes from `EngineHost`. Otherwise local dev leaks containers across restarts (REPL had this; copy that pattern).

6. **Anthropic API key.** Read from `process.env.ANTHROPIC_API_KEY`. EngineHost passes to engine when constructing.

## Out of scope (explicitly)

- The legacy worker. It stays. We don't migrate any of its routes.
- Workflows, integrations, mailbox, dashboards, OAuth, GitHub/Google/Slack/Telegram, channels, webhooks, repo providers, persona system, plugin system.
- Multi-process scaling, distributed engine state, durable WebSocket sessions, multiplayer.
- Real auth. Stub user only.
- Any CF deploy work for this package. CF stays on the old worker until parity is reached on the new API and we cut over (separate plan).
