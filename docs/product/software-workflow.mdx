Valet can act like an engineering teammate. Give it a task in your repo and it clones the code into a sandbox, works on a branch, runs checks, and opens or updates a pull request. This page walks through how that workflow actually runs so you know what to expect, what is safe, and how to recover when something goes wrong.

## How Valet works on a repo

When you ask Valet to change code, it:

1. Starts a child session with a sandbox dedicated to that repo.
2. Clones the repository into the sandbox workspace.
3. Checks out a starting branch (typically the one you specified, or the repo default).
4. Reads the relevant files and plans the change.
5. Creates a working branch for its commits.
6. Edits files, commits, and pushes the working branch to the remote.
7. Opens a pull request from the working branch into the base branch, or updates an existing one if you pointed it at one.

The orchestrator task thread shows what the child is doing. The pull request is the durable output.

## Branch and pull request policy

Valet follows a consistent branch and PR policy so its changes are easy to review:

- The branch you give Valet (or the repo default) is treated as the **base branch**.
- Valet creates or reuses a **separate working branch** for its commits. It does not commit directly to your base branch.
- Valet opens or updates a **pull request** from the working branch into the base branch.
- If you asked Valet to update an existing PR, it pushes new commits to that PR's branch rather than creating a new one.
- A task is not considered done until the branch is pushed and the PR exists (or you explicitly said no PR is needed).

If the push or PR step fails — for example, because of an auth issue or a remote conflict — Valet treats the task as incomplete and surfaces the blocker in the task thread instead of claiming success.

## CI and checks

Valet pushes to a real branch, so your normal CI runs against its commits just like a human PR. Valet can watch the results and respond to failures.

| When | What Valet does |
| --- | --- |
| Before opening a PR | Reads the files, plans the change, and runs any quick checks it can run inside the sandbox. |
| After pushing | Lets your CI run on the PR. |
| On CI failure | Reports the failure in the task thread and can attempt a fix in the same working branch. |
| On CI success | Reports the green status and waits for human review and merge. |

Valet does not bypass required checks or branch protection. If your repo requires reviews, signed commits, or status checks before merge, those rules still apply.

## Approvals on sensitive actions

Pushing code, opening a PR, and merging are sensitive operations. Valet respects your org's [approvals](/product/approvals) settings for these. Depending on configuration, Valet may ask before:

- Pushing to a remote branch.
- Opening or updating a pull request.
- Merging a pull request.
- Force-pushing or deleting a branch.

If an approval is required and you cancel it, Valet stops at that step and reports what is still pending in the task thread.

## Recovering from interrupted sessions

This is the most important rule for safe coding work in Valet:

**Push early. Un-pushed work in a child session can be lost.**

A child session is a live sandbox. If it ends — because the task finished, because it timed out, because you cancelled it, or because the platform recovered it — anything that was only on disk inside that sandbox may not come back. Once a commit is pushed to your git remote, it is safe. Until then, it is only in the sandbox.

In practice:

- Ask Valet to commit and push as soon as a piece of work is in a sensible state, not only at the end.
- If a session ends unexpectedly, check the branch on your remote first — Valet may have already pushed the work.
- If the branch is not on the remote, the change is most likely unrecoverable. Start the task again rather than waiting for the sandbox to be restored.

This is why repo work runs in [child sessions](/product/child-sessions) in the first place: the sandbox is meant to be disposable. Your source of truth is your git remote and the PR.

## What you see in the task thread

Even though the real work is in a child session, the task thread should always show:

- Which repo and base branch Valet is working on.
- The current working branch.
- The PR (number and link) once it exists.
- Progress on the change, including any CI results.
- Anything waiting on you — questions, approvals, or review.

If something is missing, ask Valet to summarize what it has done and where the work currently lives.
