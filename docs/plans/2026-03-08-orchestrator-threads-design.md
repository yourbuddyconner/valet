# Orchestrator Threads — Design Document

## Problem

The orchestrator session currently requires a full sandbox refresh to start a new conversation. This rotates the session ID, creates a new Durable Object, and discards all in-memory state. The sandbox refresh is a heavyweight escape hatch — users need a lightweight way to start fresh conversations while preserving history.

Additionally, OpenCode session history lives only in the sandbox filesystem and is lost when the sandbox dies.

## Solution

Threads map 1:1 to OpenCode sessions, giving clean context boundaries without sandbox restarts. D1 is the persistent source of truth for thread history. OpenCode sessions are ephemeral — they die with the sandbox, and that's fine.

## Design Decisions

| Decision | Choice |
|----------|--------|
| Context between threads | Orchestrator memories carry forward; each thread gets a clean conversation window |
| Thread history UI | Dedicated route (`/sessions/:sessionId/threads`), main view shows active thread only |
| Thread titles | Auto-generated by OpenCode's internal title agent on first response |
| Viewing old threads | Read-only, with a "Continue" action that starts a new thread seeded with a summary |
| Starting new threads | Explicit "New Thread" button only, no auto-split |
| OpenCode integration | Each thread = one OpenCode session via the session API |
| History persistence | Runner mirrors OpenCode session metadata + messages to D1 |
| Continuation strategy | Runner generates summary from D1-stored messages via LLM, injects into new OpenCode session |
| Backwards compatibility | None. Old messages without `thread_id` are orphaned. Clean break. |

## Data Model

### New `session_threads` table

```sql
CREATE TABLE session_threads (
  id TEXT PRIMARY KEY,
  session_id TEXT NOT NULL,
  opencode_session_id TEXT,
  title TEXT,
  summary_additions INTEGER DEFAULT 0,
  summary_deletions INTEGER DEFAULT 0,
  summary_files INTEGER DEFAULT 0,
  status TEXT NOT NULL DEFAULT 'active',
  message_count INTEGER DEFAULT 0,
  created_at TEXT NOT NULL,
  last_active_at TEXT NOT NULL
);
```

### Modification to `messages` table

```sql
ALTER TABLE messages ADD COLUMN thread_id TEXT REFERENCES session_threads(id);
```

### Key relationships

- `sessions` 1:N `session_threads` — one orchestrator session has many threads
- `session_threads` 1:N `messages` — each thread contains its messages
- `opencode_session_id` is ephemeral — set when an OpenCode session is live, nulled on sandbox restart

## API Endpoints

| Method | Endpoint | Purpose |
|--------|----------|---------|
| `GET` | `/api/sessions/:sessionId/threads` | List threads (paginated, newest first) |
| `POST` | `/api/sessions/:sessionId/threads` | Create a new thread |
| `GET` | `/api/sessions/:sessionId/threads/:threadId` | Thread detail + messages |
| `GET` | `/api/sessions/:sessionId/threads/active` | Get or create the active thread (for channel adapters) |
| `POST` | `/api/sessions/:sessionId/threads/:threadId/continue` | Start new thread seeded with summary of this thread |

### Modifications to existing endpoints

- `POST /api/sessions/:sessionId/prompt` — gains optional `threadId`. If omitted, uses the active thread. If no active thread exists, auto-creates one.
- `GET /api/sessions/:sessionId/messages` — gains optional `?threadId=` filter.

## Runner & OpenCode Integration

### Thread lifecycle

```
New Thread → Runner calls POST /session on OpenCode → gets opencode_session_id
  → User sends messages → Runner routes to that session via POST /session/:id/prompt_async
  → First response completes → Runner reads title from GET /session/:id → sends to DO → D1

New Thread again → previous OpenCode session stays in OpenCode but is no longer active
  → Runner creates new session, repeats cycle

Sandbox restart → all OpenCode sessions gone
  → D1 still has full thread history
  → Next "New Thread" starts fresh as normal
```

### Title and summary persistence

OpenCode auto-generates titles via its internal `ensureTitle` on the first assistant response (<=50 chars). The summary (additions/deletions/files changed) is updated after every assistant message/tool execution.

OpenCode publishes `session.updated` events for both. The Runner subscribes to these events and forwards changes to the DO, which persists them to D1's `session_threads` table.

### "Continue from old thread"

1. DO pulls old thread's messages from D1
2. Runner creates a new OpenCode session
3. Runner generates a compaction-style summary from old messages via LLM call (Goal, Instructions, Discoveries, Accomplished, Relevant files)
4. Summary injected as context into the new OpenCode session
5. User's message goes to the new session as normal

## WebSocket Protocol Additions

### Runner → DO

- **`thread.created`** — `{ threadId, openCodeSessionId }`
- **`thread.updated`** — `{ threadId, title?, summaryAdditions?, summaryDeletions?, summaryFiles? }`
- Existing `message.create` / `message.part.*` / `message.finalize` gain optional `threadId` field

### DO → Client (EventBusDO)

- **`thread.created`** — broadcast so UI updates thread list without polling
- **`thread.updated`** — broadcast title/summary changes for live UI updates
- Existing message events carry `threadId` so the frontend filters to active thread only

## Frontend

### Main orchestrator session view (`/sessions/:sessionId`)

- Chat panel shows the active thread. Same as today, no extra chrome.
- "New Thread" button in chat header toolbar.
- Small breadcrumb showing current thread title; clicking navigates to thread history.

### Thread history page (`/sessions/:sessionId/threads`)

- List view, newest first, showing per thread:
  - Auto-generated title
  - Relative timestamp
  - First message preview (truncated)
  - Diff stats badge: `+42 -17 across 5 files`
  - Message count
- Clicking a thread opens read-only conversation view (full messages, no input box)
- "Continue" button at bottom starts a new thread seeded with this thread's summary

### Scope

- Threads are orchestrator-only. Non-orchestrator sessions are unaffected.

## Channel Integration Surface

Threads are designed to be channel-agnostic. Any channel adapter (web, Slack, Telegram) can integrate with threads via:

- **`GET /threads/active`** — get-or-create the active thread. This is the primary entry point for routing incoming messages from any channel.
- **`POST /threads`** — explicitly create a new thread (e.g., user sends `/new` command via Slack).
- **Messages carry `channelType`/`channelId`** — each message records which channel it came from, enabling per-channel response routing.
- **The DO translates `threadId` to channel routing** — internally, `channelType='thread', channelId=threadId` routes to the correct OpenCode session regardless of which external channel the message originated from.

Slack-specific routing (DMs vs channel @mentions, user mapping, onboarding) is out of scope for this feature and will be designed separately.

## Not Building (YAGNI)

- Thread search/filtering
- Thread deletion or manual archival
- Thread merging
- Threads for non-orchestrator sessions
- Thread-level permissions or sharing
- Drag-and-drop messages between threads
- Backwards compatibility / legacy thread migration
