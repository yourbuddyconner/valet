---
title: Built-in tools
description: Reference for the sandbox and orchestrator tools every Valet session has access to before any integration is connected.
---

Every Valet session loads a fixed set of built-in tools alongside whatever [integration actions](/product/integrations/overview) are connected. These tools cover memory, task tracking, session orchestration, channel replies, workflow management, persona and skill management, secrets, browser control, and tunnels.

Built-in tools talk to the Valet platform directly through the [internal API](/engineering/runner-and-opencode#internal-api) — they do not require a connected integration. They are loaded from `/opencode-config/tools/` inside each sandbox at startup.

## Risk and approvals

Most built-in tools are low-risk reads and writes that happen against the user's own session state, memory, or task board. They do not currently flow through the same per-action approval prompt as integration actions; instead they are governed by:

- Which tools are enabled for the persona and session type. The orchestrator strips a small set of child-only tools (`complete_session`, `notify_parent`) at startup.
- The internal-API surface, which is scoped to the session's own user and org.
- Tool-level guards (for example, `spawn_session` refuses to run from a child session).

The columns below capture the call shape from code. They do **not** indicate a risk level in the sense used by [integration actions](/product/integrations/overview).

## Memory

| Tool | What it does | Key params |
| --- | --- | --- |
| `mem_read` | Read a memory file or list a directory. If the path ends with '/' or is empty, returns a directory listing. If the path is a file (e.g. 'projects/valet/repo.md'), returns its content. Use this to recall user preferences, project context, and past decisions. | `path`? |
| `mem_write` | Create or overwrite a memory file. Memories persist across conversations and sandbox restarts. Use paths to organize: preferences/, projects/<name>/, workflows/, journal/, notes/. Files under preferences/ are auto-pinned (never pruned). Writing to an existing path replaces the content and bumps the version. | `path`, `content` |
| `mem_patch` | Surgically edit a memory file without reading it first. Supports append, prepend, find-and-replace, insert_after, and delete_section. Ideal for journal entries (append), fact updates (replace), and section management. If the file doesn't exist, append/prepend will create it; other ops are skipped. | `path`, `operations` |
| `mem_search` | Search memory files using full-text search with optional LLM re-ranking. Returns the most relevant files for the query, with match-aware snippets. Use this before responding to any new request that may involve known projects, preferences, workflows, or past decisions. | `query`, `path`?, `rerank`?, `limit`? |
| `mem_rm` | Delete a memory file or all files under a directory. Use a path ending with '/' to delete all files under that prefix. Use an exact file path to delete a single file. | `path` |

### `mem_read`

Read a memory file or list a directory. If the path ends with '/' or is empty, returns a directory listing. If the path is a file (e.g. 'projects/valet/repo.md'), returns its content. Use this to recall user preferences, project context, and past decisions.

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `path` | string | No | Path to read. Examples: '' (root listing), 'preferences/' (list preferences), 'projects/valet/repo.md' (read file). Omit leading slash. |

### `mem_write`

Create or overwrite a memory file. Memories persist across conversations and sandbox restarts. Use paths to organize: preferences/, projects/<name>/, workflows/, journal/, notes/. Files under preferences/ are auto-pinned (never pruned). Writing to an existing path replaces the content and bumps the version.

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `path` | string | Yes | File path (no leading slash). Examples: 'preferences/coding-style.md', 'projects/valet/repo.md', 'notes/team.md' |
| `content` | string | Yes | The file content (markdown recommended). |

### `mem_patch`

Surgically edit a memory file without reading it first. Supports append, prepend, find-and-replace, insert_after, and delete_section. Ideal for journal entries (append), fact updates (replace), and section management. If the file doesn't exist, append/prepend will create it; other ops are skipped.

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `path` | string | Yes | File path to patch. |
| `operations` | array | Yes | Operations to apply in order. |

### `mem_search`

Search memory files using full-text search with optional LLM re-ranking. Returns the most relevant files for the query, with match-aware snippets. Use this before responding to any new request that may involve known projects, preferences, workflows, or past decisions.

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `query` | string | Yes | Search query. Examples: 'valet deployment', 'auth cloudflare', 'coding preferences' |
| `path` | string | No | Optional path prefix to scope the search. Example: 'projects/valet/' |
| `rerank` | boolean | No | Whether to re-rank results with LLM (default true, set false for speed) |
| `limit` | number | No | Max results to return after re-ranking (default 5) |

### `mem_rm`

Delete a memory file or all files under a directory. Use a path ending with '/' to delete all files under that prefix. Use an exact file path to delete a single file.

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `path` | string | Yes | Path to delete. 'notes/outdated.md' deletes one file. 'journal/' deletes all journal files. |

### `spawn_session`

Spawn a new autonomous child agent session. The child runs in its own sandbox with the given task as its initial prompt. Use this for parallel work: delegating subtasks, working on multiple repos, or fire-and-forget operations. The child inherits your environment (API keys, GitHub token, git config). Returns the child session ID which you can use with send_message and read_messages to communicate with it.

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `task` | string | Yes | The task/prompt for the child agent to work on autonomously |
| `workspace` | string | Yes | Short human-readable workspace name for the child session. Use a simple identifier like the repo name (e.g. 'my-app', 'frontend', 'api-server'). Do NOT use file paths, URLs, or Modal volume paths — just a short name. |
| `repo_url` | string | No | Git repository URL for the child to clone (optional, inherits parent's repo if not specified) |
| `branch` | string | No | Git branch for the child to check out (required for PR work) |
| `ref` | string | No | Git ref (commit SHA or tag) for the child to check out (optional). If provided, takes precedence over branch. |
| `title` | string | No | Human-readable title for the child session (defaults to workspace name) |
| `source_type` | enum(pr,issue,branch,manual) | No | Source type for the child session (defaults to parent's source type) |
| `source_pr_number` | number | No | PR number when the child works on a pull request |
| `source_issue_number` | number | No | Issue number when the child works on an issue |
| `source_repo_full_name` | string | No | Repository full name as owner/repo (e.g. 'octocat/hello-world') |
| `model` | string | No | Model ID for the child session to use (e.g. 'anthropic/claude-sonnet-4-5-20250929'). If not specified, inherits parent's model preferences. |
| `persona_id` | string | No | Persona ID to use for the child session. The persona's instructions and attached skills will be loaded at session start. |

### `send_message`

Send a message or prompt to another agent session. The message is queued as a user prompt in the target session. Use this to give follow-up instructions to child sessions you spawned, or to communicate with sibling sessions. Only works with sessions belonging to the same user.

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `session_id` | string | Yes | The target session ID to send the message to |
| `message` | string | Yes | The message content to send as a prompt to the target session |
| `interrupt` | boolean | No | If true, abort the target session's current work before delivering this message. Default: false (message is queued). |

### `read_messages`

Read messages from another agent session's conversation. By default returns the most recent messages (newest first, in chronological order). Use this to check on a child session's progress, read its results, or monitor what it's working on. Pass 'after' to paginate forward from a specific timestamp. Only works with sessions belonging to the same user.

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `session_id` | string | Yes | The target session ID to read messages from |
| `limit` | number | No | Maximum number of messages to return (default 20) |
| `after` | string | No | ISO timestamp cursor — only return messages after this time (for pagination) |

### `forward_messages`

Forward messages from another session into your current chat. This copies the messages directly into your conversation so the user can see them — without you having to read and retype them. Use this after read_messages when you want to share a child session's output with the user. The messages appear as quoted blocks attributed to the source session.

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `session_id` | string | Yes | The source session ID to forward messages from |
| `limit` | number | No | Maximum number of messages to forward (default 20) |
| `after` | string | No | ISO timestamp cursor — only forward messages after this time (for pagination) |

### `get_session_status`

Get the current status of another agent session, including its recent messages. Use this to check on child sessions' progress, see what they're working on, or determine if they're done. Key statuses: running (active), idle (sandbox alive but agent idle), hibernated (sandbox stopped — not running, no processes alive), terminated (ended permanently). If runnerConnected is false, the sandbox is not running. Only works with sessions belonging to the same user.

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `session_id` | string | Yes | The target session ID to get status for |

### `list_sessions`

List child sessions spawned by the current session. Returns each session's ID, title, status, workspace, and PR info. Use this to see what sessions you've spawned, check their statuses, and find session IDs for further inspection with get_session_status or read_messages.

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `_placeholder` | string | No | Unused |

### `terminate_session`

Terminate a child session that you previously spawned. Use this to stop a child agent that is no longer needed, stuck, or has gone off track. You can only terminate sessions that are direct children of your own session.

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `session_id` | string | Yes | The session ID of the child session to terminate (returned by spawn_session) |

### `wait_for_event`

Yield control and wait for the next incoming event. Call this when you have finished your current work and are waiting for something external — a child session's notify_parent, a user message, or any other prompt. Your turn ends immediately after this tool returns. The next message you receive will be from whoever wakes you (child notification, user, etc.). Prefer this over sleep loops when the wait time is unknown or when you expect a child to notify you proactively.

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `reason` | string | No | Brief note about what you're waiting for (e.g. 'Waiting for child session to complete') |
| `session_ids` | array | No | Optional list of child session IDs to monitor. If omitted, all child sessions are monitored. |
| `notify_on` | enum(terminal,status_change) | No | Which events should wake you. 'terminal' (default) only fires when a child reaches a terminal status. 'status_change' fires on any child status transition. |
| `statuses` | array | No | Optional list of statuses to trigger on (e.g. ['terminated','error']). Overrides notify_on. |

### `sleep`

Pause execution for a specified duration. Use this when you need to wait before checking on a child session's progress, poll for results, or give a task time to complete. This keeps the agent loop alive (preventing idle timeout) while waiting.

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `seconds` | number | Yes | Number of seconds to sleep (1-300) |
| `reason` | string | No | Why you are sleeping (shown to the user while waiting) |

### `notify_parent`

Send a message to your parent session (the orchestrator or session that spawned you). Use this to report progress, deliver results, ask for clarification, or flag blockers. The message is queued as a user prompt in the parent session. Only available in child sessions — orchestrators and top-level sessions do not have a parent.

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `message` | string | Yes | The message to send to your parent session |
| `interrupt` | boolean | No | If true, abort the parent's current work before delivering. Default: false (message is queued). |

### `complete_session`

Signal that your assigned task is fully complete and shut down this session. Call this when you have finished all work, committed your changes, and have nothing left to do. This will terminate your sandbox — only call it when you are truly done.

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `_placeholder` | string | No | Unused |

### `channel_reply`

Send a reply to a specific channel (e.g. Telegram, Slack). Use this when a user's message came from an external channel (indicated by [via <channelType> \| chatId: <id>] prefix) and you want to respond on that same channel. The response will be delivered directly to the user on the originating platform. You can optionally attach an image or file.

| Param | Type | Required | Description |
| --- | --- | --- | --- |
| `channel_type` | enum(telegram,slack) | Yes | The channel type to reply on (e.g. 'telegram') |
| `channel_id` | string | Yes | The channel/chat identifier (e.g. Telegram chatId from the message prefix) |
| `message` | string | Yes | The message text to send back on the channel (can be empty when sending an image) |
| `image_path` | string | No | Absolute path to an image file to send (PNG, JPEG, GIF, WebP) |
| `file_path` | string | No | Absolute path to a file to attach (any type: PDF, CSV, ZIP, etc.) |
| `file_name` | string | No | Override the filename shown to the recipient (defaults to the basename of file_path) |
| `follow_up` | boolean | No | Controls whether this reply clears the follow-up reminder timer. Defaults to true. IMPORTANT: For most replies, you do NOT need to set this — just omit it and the default (true) will mark the conversation as handled. Only set follow_up=false when you are sending a brief acknowledgment AND you plan to do async/deferred work (spawning a child session, long research, etc.) before sending a real answer later. If your reply IS the answer (even a short one like 'Hey, how can I help?'), leave follow_up unset or set it to true so the reminder timer is cleared. |

### `transcribe_audio`

Transcribe an audio file to text using whisper.cpp (local speech-to-text). Accepts common audio formats: wav, mp3, ogg, webm, flac, m4a, mp4. Use the 'base' model (default) for fast transcription (~3-5s for 30s audio) or 'large' for higher accuracy.

| Param | Type | Required | Description |
| --- | --- | --- | --- |
