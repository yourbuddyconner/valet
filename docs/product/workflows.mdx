A workflow is a repeatable way to ask Valet to do work. Use one when the same kind of task should happen more than once, follow a consistent process, or start automatically.

<MascotNote
  title="Reference"

>
  Write a workflow like a checklist for a coworker: what starts it, what information to gather, what to produce, and when to stop for review.
</MascotNote>

## When to create a workflow

Create a workflow for recurring work such as:

- Daily queue triage.
- Weekly team summaries.
- Customer follow-up after a form or support event.
- Release-prep checklists.
- Issue or pull-request review.
- Research that always needs the same inputs.

If the task is one-off or still unclear, start with a normal [task](/product/tasks). Turn it into a workflow after the pattern repeats.

## Create a workflow

<Steps>
  <Step title="Name the job">
    Use a name that describes the outcome, such as "Weekly support summary" or "Triage new pull requests."
  </Step>
  <Step title="Choose how it starts">
    Decide whether a person runs it, a schedule runs it, or another system sends Valet an event.
  </Step>
  <Step title="Write the instructions">
    Tell Valet what to check, which tools to use, what good output looks like, and what should be ignored.
  </Step>
  <Step title="Add review points">
    Mark the steps that need approval before Valet writes, sends, deletes, merges, or changes something important.
  </Step>
  <Step title="Run it once">
    Test the workflow on a small, real example. Review what Valet did and adjust the instructions.
  </Step>
</Steps>

## A simple example

```text
Workflow: Weekly customer follow-up

Start: every Friday morning

Instructions:
- Review open customer issues in Linear.
- Find accounts with stale follow-up.
- Draft a short update for each account.
- Ask for approval before sending anything externally.
- Post a summary in Slack when finished.
```

## Improve the workflow

After a few runs, review:

- Which steps were unclear.
- Which approvals were useful or noisy.
- Which connected tools were missing.
- Whether the output was easy to use.
- Whether the workflow should be split into smaller workflows.
