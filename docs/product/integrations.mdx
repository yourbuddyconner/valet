# Integrations

Integrations let agents work with external systems. The agent discovers actions with `list_tools` and invokes them with `call_tool`. Each call still passes through Valet's policy and approval system.

## Integration model

Valet has two integration paths:

- **First-party action plugins** compile into the Worker and implement actions directly.
- **MCP-backed plugins** proxy remote MCP services and expose their tools through Valet.

Credentials are stored encrypted and resolved at request time. OAuth-backed integrations can refresh tokens when supported by the provider path.

## Agent invocation flow

Agents should use the integration model like this:

1. Call `list_tools` to discover available services and actions.
2. Choose the action by service and action ID.
3. Call `call_tool` with the action input.
4. Let Valet resolve action policy before execution.
5. Handle approval, denial, or execution result.

Useful tools discovered with `list_tools` should be written to memory so future sessions do not need to rediscover the same catalog from scratch.

## Credential model

Credential storage depends on the provider path:

- Login OAuth tokens for providers such as GitHub and Google live in encrypted D1 rows.
- Integration credentials are encrypted and resolved by service at request time.
- Custom MCP connectors can use no auth, OAuth, API key, or bearer token configuration.
- 1Password credentials can be injected into sandbox sessions through the configured user credential path.

Agents do not receive raw credentials directly. They call actions, and Valet resolves credentials server-side or through sandbox environment assembly where appropriate.

## GitHub

GitHub has a dedicated production path in addition to the generic integration framework. It supports OAuth login, repository listing and validation, issue and PR lookup, PR creation, org repository management, and webhook handling for pull request and push events.

GitHub session state tracks source repository, source branch, working branch, PR number, PR state, PR URL, and commit count. The sandbox can also receive `GITHUB_TOKEN`, git identity, repo URL, branch, and ref through environment assembly.

## Google Workspace

Google Workspace actions cover Gmail, Calendar, Drive, Docs, and Sheets. Gmail and Calendar include token refresh behavior and full read/write action sets. Workspace document and spreadsheet tools are packaged under the Google Workspace plugin.

Gmail write actions accept Markdown bodies, render semantic HTML, and send multipart email with the original Markdown as the plain-text fallback.

## Custom MCP connectors

Admins can configure custom remote MCP connectors. These can use no auth, OAuth with admin-provided client credentials, API keys, or bearer tokens depending on connector configuration.

The Worker validates outbound connector URLs, builds request-scoped connector context, and clears cached tool entries when connector URL, auth, status, or header configuration changes.

## MCP-backed plugins

MCP-backed plugins use Valet's MCP client to initialize a remote MCP session, list tools, and call tools. OAuth-backed MCP services use MCP-scoped tokens, so token validation and refresh are handled differently from traditional provider APIs.

## Sync framework

Some first-party integrations also have a generic sync framework:

- A D1 `integrations` row records user, service, status, config, and sync metadata.
- A sync log records each sync attempt.
- Concrete integration classes implement credential validation, connection test, sync, fetch, push, and webhook methods.

The production GitHub path uses dedicated routes and services for repository and PR behavior even though a generic GitHub integration class also exists.
