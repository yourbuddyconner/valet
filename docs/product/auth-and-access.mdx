# Auth and Access

Valet separates account authentication, organization gates, session access, and sandbox iframe access. A user can be allowed into the app, but still need session-level permissions and short-lived sandbox tokens to open specific workstations.

## Login

Valet supports OAuth login through GitHub and Google. The flow is:

1. The browser starts `GET /auth/{provider}`.
2. The Worker creates a short-lived state JWT.
3. The provider redirects back with `code` and `state`.
4. The Worker verifies state, exchanges the code, fetches the provider profile, and creates or updates the Valet user.
5. The Worker stores a hashed session token in `auth_sessions` and redirects the browser with the plaintext token once.

The first user created in an empty deployment is automatically promoted to admin.

## App tokens

Valet accepts bearer tokens from:

- `auth_sessions`: browser login tokens with a seven-day expiry.
- `api_tokens`: personal programmatic tokens with optional expiry and soft revocation.

The auth middleware checks session tokens first, then API tokens. On success, routes receive the authenticated user ID, email, and role.

## Organization gates

Signup can be restricted by:

- Existing user status.
- Invite code.
- Allowed email domain.
- Explicit email allowlist.
- Email-specific invite.
- Legacy `ALLOWED_EMAILS` environment variable.

If no gate is configured, any authenticated provider account can be allowed.

## Roles and admin access

Users have `admin` or `member` role. Admin-only routes use a simple role check after authentication. Admins manage org repositories, org settings, provider keys, custom providers, custom MCP connectors, and org-level policies.

## Session access

Session participation uses three roles:

- **Owner** can update and terminate the session.
- **Collaborator** can participate.
- **Viewer** can inspect.

Access can come from ownership, participant rows, org visibility settings, or share-link redemption. Share links carry a token, granted role, optional expiry, optional use limit, and active flag.

## Sandbox access

Opening VS Code, VNC, terminal, or custom tunnel routes requires a sandbox JWT from `GET /api/sessions/:id/sandbox-token`. The token lasts 15 minutes and is validated by the Runner gateway before the gateway sets its own short-lived session cookie.
